# pdfify — Design & Architecture

## Overview

pdfify is a Go CLI tool that converts Markdown documents to professionally styled PDFs. It wraps pandoc + XeLaTeX + mermaid-cli inside a Docker container, providing a zero-dependency experience for the user (just Docker + the pdfify binary).

The core problem: generating good-looking PDFs from Markdown requires a complex toolchain (pandoc, LaTeX distribution, mermaid, Chromium for rendering). Rather than asking users to install all of this, pdfify manages a Docker image containing everything.

The original implementation was a 1300-line bash script. This Go rewrite provides proper argument parsing, validation, themes, a preferences system, watch/edit modes with browser UI, self-update, and unit tests.

## Architecture

```
User --> [Cobra CLI] --> [Config Merger] --> [Converter Pipeline]
                              |                     |
                    [preferences.yaml]    [Docker Container]
                    [frontmatter]              |
                    [CLI flags]         [bash script]
                                             |
                                    [callout processing]
                                    [mermaid rendering]
                                    [page break injection]
                                    [pandoc + XeLaTeX]
                                             |
                                           [PDF]
```

For `--watch` and `--edit` modes:
```
User --> [Cobra CLI] --> [Converter] --> [HTTP Server]
                              ^               |
                              |         [SSE push reload]
                              |         [PDF iframe]
                              |         [CodeMirror editor]
                              |               |
                         [file watcher]  [Browser UI]
```

## Key Design Decisions

1. **Docker for conversion tools** — The LaTeX + pandoc + mermaid + Chromium toolchain is ~2GB installed. Docker isolates this complexity. The image builds once and is cached. Smart inspection (via Dockerfile hash label) detects when rebuilds are needed.

2. **Three-tier configuration** — Settings cascade: CLI flags > YAML frontmatter > `~/.config/pdfify/preferences.yaml` > built-in defaults. This lets users set global preferences (author, theme) while overriding per-document via frontmatter.

3. **Bash script inside Docker** — The actual markdown preprocessing (callouts, mermaid, page breaks) happens in a bash script generated by Go and mounted into the container. This avoids complex Docker API usage and keeps the pipeline debuggable.

4. **Theme abstraction** — All visual parameters (colors, fonts, heading styles, mermaid palettes) are defined in theme structs. The LaTeX preamble is generated from theme values, not hardcoded. Adding a theme means adding a function that returns a `Theme` struct.

5. **SSE for watch/edit modes** — Server-Sent Events (not WebSocket) for reload notifications. Simpler, auto-reconnecting, and sufficient for one-way push. The browser UI is embedded via `//go:embed`.

6. **Auto-detection of heading structure** — If a document has exactly one H1, it's treated as the document title (promoted to title banner, numbered from H2). Multiple H1s means they're sections (numbered from H1).

7. **CGO_ENABLED=0 always** — Single static binary, no C dependencies. Works on any platform.

## Config System

The `Config` struct uses zero values as "not set" markers. The `Merge()` function layers two configs, with non-zero overlay values winning:

```
DefaultConfig() <-- LoadPreferences() <-- ExtractFrontmatter() <-- CLI flags
```

Boolean fields like `NumberSections` use `*bool` to distinguish "not set" (nil) from "set to false".

## Theme System

Themes define:
- Color palette (accent, headings, code blocks, callouts, tables)
- Font choices (main, mono, heading)
- Heading style (letter spacing, small caps)
- Mermaid chart appearance (colors, palettes)

All theme values flow into the LaTeX preamble via `GeneratePreamble()`.

## Docker Image Management

The Docker image is labeled with a SHA-256 hash of the Dockerfile content:
- On each run, compare the embedded Dockerfile hash with the image label
- If different, rebuild automatically (no manual `--rebuild` needed for version updates)
- If no label (pre-Go image), prompt rebuild

## Conversion Pipeline

1. **Parse config** — merge CLI + frontmatter + preferences
2. **Analyze document** — count H1s, images, mermaid blocks, callouts
3. **Generate script** — bash script for Docker with theme-aware preamble
4. **Mount and run** — Docker container: input dir (ro) + output dir (rw)
5. Inside Docker:
   - Strip H1 if promoting to title
   - Convert Obsidian callouts to LaTeX environments
   - Inject page breaks before top sections
   - Render mermaid blocks to PNG
   - Strip frontmatter (pdfify handles it, not pandoc)
   - Run pandoc with XeLaTeX, preamble, and Lua filter

## Watch/Edit Server

A lightweight HTTP server (stdlib `net/http`) with:
- `/` — HTML page (watch or edit mode)
- `/pdf` — serves the latest PDF binary
- `/events` — SSE endpoint pushing "reload" events
- `/api/content` — GET markdown content (edit mode)
- `/api/save` — POST to save content back (edit mode)

Edit mode loads CodeMirror from CDN with yaml-frontmatter mode, auto-saves on debounced change, and triggers reconversion + SSE reload.

## Testing Strategy

- **Config tests** — merge logic, validation, edge cases
- **Frontmatter tests** — extraction, injection, malformed YAML, missing delimiters
- **Theme tests** — all themes have required fields, mermaid config is valid JSON
- **Preamble tests** — generated LaTeX contains expected packages, colors, sections
- **Document analysis tests** — H1 counting, callout detection, code block handling
- **UI tests** — size formatting

Integration tests (requiring Docker) are not in the unit test suite.

## Known Limitations & Future Work

- **No HTML output** — the edit mode HTML preview is a simple client-side approximation, not full pdfify rendering
- **No sigstore verification on update** — the updater downloads but doesn't verify signatures yet (framework is in place)
- **Single-threaded conversion** — files are converted sequentially, not in parallel
- **CDN dependency for edit mode** — CodeMirror is loaded from cdnjs.cloudflare.com (could be embedded)
- **No PDF preview in terminal** — watch/edit modes require a browser
